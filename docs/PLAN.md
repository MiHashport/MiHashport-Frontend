# MiHashport Frontend — Implementation Plan

This document splits the frontend's implementation into 6 phases. Each phase
should be demoable on its own, and each checklist item is scoped to be a
single GitHub issue. Maintainers create issues from this plan; see
[CONTRIBUTING.md](../CONTRIBUTING.md) for how to pick up work.

Context: MiHashport lets people chat, build, and pay on Stellar entirely
inside WhatsApp. The conversational experience itself lives in
[MiHashport-Backend](https://github.com/MiHashport/MiHashport-Backend), which
talks to WhatsApp and to [MiHashport-Contract](https://github.com/MiHashport/MiHashport-Contract).
This frontend is the companion web app: the surface for things WhatsApp can't
do well — linking a number to a wallet, reviewing balances and transaction
history, reviewing/approving contract deployments, and giving the project a
public face.

---

## Phase 1 — Foundation

**Goal:** A working, deployable Next.js app with the project's structure, tooling, and docs in place.

**Scope:**
- Next.js (App Router, TypeScript, Tailwind) scaffold
- Base folder structure (`components/ui`, `components/features`, `lib`, `hooks`, `types`, `config`)
- Minimal landing page
- CI (build check), env scaffolding, contributor docs

**Checklist:**
- [x] Scaffold Next.js app with TypeScript, Tailwind, App Router, `src/` layout
- [x] Add base folder structure and `.env.example`
- [x] Add CI workflow (install + build)
- [x] Write README, CONTRIBUTING, AGENTS, issue/PR templates
- [ ] Set up a shared `lib/api-client.ts` for calling the backend (base URL from `src/config/env.ts`, typed fetch wrapper, error handling)
- [ ] Add a basic app shell layout (header with logo/wordmark, footer with links to sibling repos)
- [ ] Set up a `types/` module for shapes shared across features (User, WalletAccount, Transaction) as placeholders to be filled in by later phases

**Dependencies:** None — this is the starting point.

**Definition of done:** `npm run build` passes in CI on every PR to `main`; app renders the landing page; docs are accurate for a first-time contributor.

## Phase 2 — Design System & Core UI Components

**Goal:** Build the reusable UI primitives every later feature will be built from.

**Scope:**
- `src/components/ui`: buttons, inputs, cards, modals/dialogs, badges, skeleton/loading states, toasts
- Layout primitives: page container, responsive nav, mobile menu
- Light/dark theme support (Tailwind `dark:` variants already used in Phase 1)

**Checklist:**
- [ ] Button component (variants: primary/secondary/ghost/destructive, loading state)
- [ ] Input/Textarea/Select form primitives with label + error states
- [ ] Card and Modal/Dialog primitives
- [ ] Toast/notification system for success/error feedback
- [ ] Skeleton loaders and empty-state components
- [ ] Responsive nav bar + mobile menu
- [ ] Storybook-less visual smoke test page (e.g. `/dev/components`) gated out of production build, or removed before Phase 6

**Dependencies:** Requires Phase 1's folder structure and app shell.

**Definition of done:** Every component is used at least once in the app (no orphaned components); components are typed, accessible (keyboard nav, ARIA labels), and themeable.

---

## Phase 3 — Wallet Linking & Auth

**Goal:** Let a user connect their WhatsApp-linked MiHashport account to the web app and reach an authenticated dashboard shell.

**Scope:**
- Number/account linking flow (e.g. one-time code generated by the WhatsApp bot, entered on the web app)
- Session handling against the backend's auth endpoints
- Protected route layout (`(dashboard)` route group) with redirect-to-login

**Checklist:**
- [ ] Design the linking flow UI (enter phone number → enter OTP sent via WhatsApp → confirm)
- [ ] Implement API calls to backend auth endpoints (request code, verify code, refresh session)
- [ ] Store session token securely (httpOnly cookie via a Next.js route handler, not localStorage)
- [ ] Add a protected layout/middleware that redirects unauthenticated users to the linking flow
- [ ] Add a logout action
- [ ] Add error/empty states for expired or invalid codes

**Dependencies:** Requires backend endpoints for phone-code auth (MiHashport-Backend). Requires Phase 2 form components.

**Definition of done:** A user can link a WhatsApp-registered number, land on an authenticated (empty) dashboard, and log out; session persists across page reloads.

---

## Phase 4 — Wallet Dashboard

**Goal:** Give a linked user visibility into their Stellar wallet: balances, transaction history, and simple send/swap actions mirrored from WhatsApp.

**Scope:**
- Balances view (native XLM + any tracked assets)
- Transaction history list with status (pending/success/failed) and Stellar Explorer links
- Send/receive/swap forms that call backend endpoints (which submit to Stellar)

**Checklist:**
- [ ] Balances card(s) on the dashboard home, reading from backend wallet endpoint
- [ ] Transaction history list with pagination/infinite scroll
- [ ] Transaction detail view (amount, memo, hash, explorer link, network — testnet/mainnet aware)
- [ ] Send form (recipient, amount, asset, memo) wired to backend submit endpoint
- [ ] Swap form (asset in/out, amount, quoted rate) wired to backend
- [ ] Receive view (address/QR code for the linked wallet)
- [ ] Loading/error/empty states for all of the above using Phase 2 primitives

**Dependencies:** Requires Phase 3 auth. Requires backend wallet/transaction endpoints (MiHashport-Backend), which in turn rely on deployed contracts where applicable (MiHashport-Contract).

**Definition of done:** A linked user can view balances and history, and successfully submit a send and a swap on testnet from the web app.

---

## Phase 5 — Contract Deployment & Management UI

**Goal:** Give users/developers a web UI to view and interact with the Soroban contracts MiHashport deploys on their behalf.

**Scope:**
- List of contracts deployed for the current user (from WhatsApp "deploy contract" flow)
- Contract detail view: contract ID, network, deployment status, ABI-derived method list
- Simple method-invocation forms for read/write contract calls

**Checklist:**
- [ ] Contracts list view (per-user deployed contracts, status, network)
- [ ] Contract detail page showing contract ID, deploy tx, explorer link
- [ ] Dynamic method-invocation form generated from contract spec/ABI returned by backend
- [ ] Read-only ("simulate") call support vs. signed write-call support, clearly distinguished in the UI
- [ ] Error surfacing for failed simulations/invocations (map Soroban error enums to readable messages)
- [ ] Empty state for users with no deployed contracts yet, linking back to the WhatsApp flow

**Dependencies:** Requires Phase 3 auth and Phase 4's API-calling patterns. Requires MiHashport-Contract to expose stable contract specs, and MiHashport-Backend endpoints for listing/invoking a user's deployed contracts.

**Definition of done:** A user with a deployed contract can view it and successfully simulate and invoke at least one method from the web app.

---

## Phase 6 — Polish, Observability & Launch Readiness

**Goal:** Take the app from "feature complete" to something safe to point real users and mainnet traffic at.

**Scope:**
- Cross-cutting UX polish, accessibility pass, mobile responsiveness pass
- Error boundaries, monitoring/analytics, performance budget
- Mainnet-vs-testnet safeguards in the UI

**Checklist:**
- [ ] Global error boundary + Next.js `error.tsx`/`not-found.tsx` pages styled to match the app
- [ ] Accessibility pass (keyboard navigation, color contrast, screen reader labels) across Phases 2–5 components
- [ ] Mobile responsiveness pass on dashboard, send/swap, and contract views
- [ ] Network-mismatch guard (visible banner + blocked actions if `NEXT_PUBLIC_STELLAR_NETWORK` doesn't match the account's actual network)
- [ ] Basic analytics/telemetry hook for key actions (link account, send, swap, deploy) behind an opt-out
- [ ] Lighthouse/perf pass on the landing page and dashboard shell
- [ ] Production deployment docs (env vars, hosting target) added to README

**Dependencies:** Requires Phases 2–5 to be functionally complete.

**Definition of done:** App is responsive and accessible end-to-end, mainnet/testnet mismatches are impossible to trigger silently, and README has accurate production deployment instructions.
